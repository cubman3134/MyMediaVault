## Route fixups vs the plan skeleton

The plan skeleton's key sequence was the SHAPE; the real themed-XMB layout was mapped with
`uitest.py state` first and encoded in `perfbaseline.py` with inline comments. Deviations:
1. Reach Games with a single `right` (horizontal=category, vertical=item), not `down 2; right 3`.
2. Enter the Games rail default **Recent** (local ROMs, Jurassic Park at idx 0) rather than a
   console grid — Recent is the proven reliable local-launch path (Tasks 2/3); a console tile is not
   guaranteed locally launchable.
3. Launch needs TWO enters: Enter opens a per-game **NavMenu** overlay ("Play" preselected), a second
   Enter triggers Play (page flips to `RetroView`). The script verifies both stages and records
   `open-game` SKIPPED if the overlay never appears or the core never reaches RetroView.
4. A SINGLE `escape` exits the core straight back to themed home (no esc-menu confirm to enter).
5. Do not spam `escape 3` at the end; instead escape only while still in RetroView (max 3) so every
   run ends on the same screen (Games / Recent), keeping runs comparable.
6. The 50-row scroll is symmetric (down 50 / up 50) so the selection always returns to idx 0
   (Jurassic Park) before the open step, regardless of Recent's length.

## Verdicts

The two whales were fixed and their commits are named here so the exit-criteria audit is complete:
**startup.total 6800→1492** via commit **5c14e4a** (theme UI sounds opened off the GUI thread; the
`startup.firstpaint` true-paint guard was shipped alongside it), and **thumbs.page warm→0** via commit
**cd05b37** (browsed posters disk-cached so warm visits skip the fetch). This section closes the
loop on the remaining span tail. Each verdict is grounded in the code at the span's boundary on the
current build (`perf/track-plan1`, HEAD `cd05b37`). "At floor" means the cost lives inside a boundary
the perf track does not cross (mpv / libretro core / Qt-QML internals / network round-trip); the
instrumented, in-our-code portion is already thin.

- **startup.settings** — AT FLOOR (and NOT a sync-network stall). Task 2's note said the phase ends
  right after `Achievements::tryLoginWithStoredToken()`; that function is **already async**. It does two
  synchronous `store().value("ra/user"|"ra/token")` QSettings reads, then calls
  `rc_client_begin_login_with_token`, whose HTTP goes through `serverCallCb` (native/src/core/Achievements.cpp:75)
  which posts via `QNetworkAccessManager` and handles the reply on the `finished` signal — no blocking
  wait on the GUI thread, so this is NOT the unbounded-stall class of the fixed audio bug. Login still
  happens; RA badges still appear on the async reply; failure stays silent/notified exactly as before.
  The span (MainWindow.cpp:186–243) actually brackets construction of MpvWidget / RetroView (SDL gamepad
  enumeration) / EbookView / PdfView / ComicView / TraktClient / CastManager / SubtitleFetcher — subsystem
  widget init. That is the 124–199 ms, it is noisy first-run disk-cache jitter (84→199 across baseline
  runs; 124 on re-measure), and it is Qt-widget-construction floor, not our code doing anything dumb.
- **startup.home** — AT FLOOR (Qt/QML floor). Span = `showHomeScreen()` (MainWindow.cpp:770–772), the
  first build+show of the themed QML home (or classic HomeView). 173–183 ms is one-time widget/QML scene
  instantiation; nothing sync-I/O or redundant sits before the Qt boundary.
- **startup.addons** — AT FLOOR (thin by design). 4 ms. Span = AddonManager ctor's initial disk reload
  + addon-view wiring (MainWindow.cpp:247–265). Sub-5 ms is jitter, not signal.
- **startup.theme** — AT FLOOR (thin by design). 3 ms. Span = BackgroundMusic ctor + theme/BGM signal
  wiring (MainWindow.cpp:336–349). The previously-fixed sync UI-sound decode was already moved off this
  path (commit 5c14e4a).
- **catalog.load** — AT FLOOR (network round-trip). Span = begin at `mgr_->requestCatalog/requestDetail`
  dispatch → end after `populate()` in `onCatalogReady` (HomeView.cpp:3433–3464). The 93–112 ms is the
  addon backend request/response round-trip; `populate()` (our code) is the small tail. Addon-network floor.
- **open.video** — AT FLOOR (mpv floor). begin at `openVideoPath`/`playStream` (MainWindow.cpp:1420,1728)
  → end in the `MpvWidget::fileLoaded` handler (MainWindow.cpp:209). The 258 ms (follow-up run) is mpv
  demuxing/opening the file; we measure up to the fileLoaded signal, never inside libmpv.
- **open.audio** — AT FLOOR (mpv floor). begin at `openAudioPath`/`openAudioStream` (MainWindow.cpp:1460,1756)
  → end in the same `fileLoaded` handler. 48 ms is mpv open of the first queued track. (The prior
  fixed bug here was the *sync sidecar/album-art fetch* before play — that path is already off-thread.)
- **open.game** — AT FLOOR (libretro floor). begin at `openGamePath` → end in `GameLauncher::open` right
  after `retro_->openGame` (GameLauncher.cpp:239) / after `emu_->play` for external (GameLauncher.cpp:427).
  79–84 ms is core `dlopen` + `retro_load_game` (ROM read + core init); measured to core load, never inside.
  CAVEAT (future-fix candidate, not in the measured data): for BIOS-needing systems (3DO/Saturn/PSX),
  `CoreManager::ensureBios` runs a SYNCHRONOUS QEventLoop download (CoreManager.cpp:186-188) inside this
  span on first launch — the same sync-network-stall class fixed for startup audio. No-op on the NES route.
- **open.reader** — UNMEASURED LIVE (span in place, no fix/verdict possible from data). begin at
  `openDocumentPath` → end after each reader-type success branch (MainWindow.cpp:1782–1815, pdf/cbz/epub).
  The standard route never opens a document, so no sample exists. The instrumentation is correct and
  orphan-safe; a future route that opens a PDF/EPUB/CBZ will populate it. Expected class: the reader
  library's parse/open (Qt PDF / ebook / comic-zip) — a boundary cost — but that is a prediction, not a
  measurement, and is flagged as such.
- **nav.select** — AT FLOOR, with a corrected diagnosis. Re-measured once on the current (warm-thumbs)
  build: **worst 429 ms, median ~1–2 ms, 73 samples**; thumbs.page fired **0 times** (posters now
  disk-cached, so the empty→non-empty thumb-queue transition never happens). The prior baseline assumed
  the ~400 ms worst was thumbs-coupled (the one paint that triggered a fresh thumbs page). **That was
  wrong: the warm cache did NOT pull the worst down (404→429).** The 429 ms is the **first** nav.select
  of the run (log line 1); every one of the other 72 presses is ≤34 ms. It is the first cold resolution
  in `requestThemedMeta` — first themed-QML meta-delegate instantiation + first `GamelistStore::lookup`
  (gamelist.xml parse) + first `MetaCache::loadArt` disk read for a freshly-entered rail, all then cached
  in `themedArtCache_` so it never recurs in-session. That is a one-shot QML/first-item warmup (Qt-QML +
  first-disk-read boundary), overlapping the still-settling home, not a repeatable per-press stall — at
  floor. Not a future-fix candidate: pre-warming the QML delegate to shave a one-time 0.4 s risks changing
  paint ordering for no steady-state gain.
